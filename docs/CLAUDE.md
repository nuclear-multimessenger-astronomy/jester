## Building Documentation

### Basic Build Commands

```bash
# Build documentation locally
uv run sphinx-build docs docs/_build/html

# Open in browser
open docs/_build/html/index.html      # macOS
xdg-open docs/_build/html/index.html  # Linux

# Build with auto-reload (requires sphinx-autobuild)
uv run sphinx-autobuild docs docs/_build/html
# Then visit http://127.0.0.1:8000

# Strict mode (same as CI/CD - fails on warnings)
uv run sphinx-build -W --keep-going docs docs/_build/html
```

**Always test in strict mode before committing** - CI/CD will fail if there are any warnings.

### Installation

Documentation dependencies are in the `[docs]` extra:
```bash
uv sync --extra docs
```

## Documentation Architecture

### Hybrid Approach

JESTER uses a **hybrid documentation system**:

1. **Auto-generated** - YAML configuration reference from Pydantic schemas
   - Source: `../jesterTOV/inference/config/schema.py`
   - Generator: `../jesterTOV/inference/config/generate_yaml_reference.py`
   - Output: `inference_yaml_reference.md`
   - **When to regenerate**: After modifying `schema.py`
   - **Command**: `uv run python -m jesterTOV.inference.config.generate_yaml_reference`

2. **Manual** - Narrative documentation, guides, examples
   - All other `.md` and `.rst` files
   - Must be updated manually when features change

## Formatting Standards

### Math in Docstrings

**Always use reStructuredText math formatting in docstrings:**

```python
def gamma_function(x: float) -> float:
    r"""Compute the gamma function.

    The gamma function is defined as :math:`\Gamma(x) = \int_0^\infty t^{x-1} e^{-t} dt`.

    For display math, use:

    .. math::
        \Gamma(x) = \int_0^\infty t^{x-1} e^{-t} dt

    Args:
        x: Input value where :math:`x > 0`
    """
```

**Key rules:**
- Use raw strings `r"""` for docstrings with LaTeX (avoids Python escape warnings)
- Inline math: `:math:\`expression\``
- Display math: `.. math::` directive
- See `../jesterTOV/eos/base.py` for reference examples

### MyST Markdown

**Cross-references** (use MyST syntax, NOT standard markdown links for internal refs):

```markdown
<!-- Referencing a section with an anchor -->
See {ref}`config-system` for details.

<!-- Creating an anchor (use explicit MyST style) -->
(config-system)=
## Configuration System
```

**Mermaid diagrams**:

````markdown
```{mermaid}
graph TD
    A[Config] --> B[Parser]
    B --> C[Sampler]
```
````

**NOT** `` ```mermaid`` (plain code fence) - must use MyST directive syntax.

## Common Documentation Tasks

### Adding New Documentation

1. Create `.md` file in `docs/`
2. Add to appropriate `toctree` in `index.rst`:
   ```rst
   .. toctree::
      :maxdepth: 2
      :caption: Section Name

      new_document
   ```
3. Build locally to verify
4. Run strict build: `uv run sphinx-build -W --keep-going docs docs/_build/html`

### Fixing Sphinx Warnings

**Common warning types:**

1. **Undefined reference**: `WARNING: undefined label: 'section-name'`
   - Add explicit MyST anchor: `(section-name)=`
   - Use {ref} syntax: `{ref}\`section-name\``

2. **Citation not found**: `WARNING: Citation [citation1] is not referenced`
   - Add citation to `../jesterTOV/references.bib` or relevant docstring
   - Use correct citation format in docstrings

3. **Toctree reference not found**
   - Ensure file exists and is in correct location
   - Check filename matches exactly (case-sensitive)

4. **Ambiguous reference**
   - Add to `nitpick_ignore` in `conf.py` if spurious
   - Or make reference more specific

### Updating Configuration Documentation

**When you modify `../jesterTOV/inference/config/schema.py`:**

1. Make your changes to Pydantic models
2. Regenerate YAML reference:
   ```bash
   uv run python -m jesterTOV.inference.config.generate_yaml_reference
   ```
3. Review changes in `inference_yaml_reference.md`
4. Update manual docs if the change is user-facing:
   - `inference.md` - Conceptual explanations
   - `inference_quickstart.md` - If it affects quick start workflow
5. Commit both `schema.py` and `inference_yaml_reference.md` together

**Recommended**: See `inference_documentation_guide.md` for complete maintenance workflow.
